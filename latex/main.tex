\begin{abstract}
大语言模型生成的控制程序在资源受限 MCU 上缺乏既紧凑又可安全执行的表示。本文提出 M-IR + M-bytecode + MVM 两层框架：M-IR 面向 LLM 生成（输出 token 显著低于 JSON，以更长的 system prompt 为代价），编译为传输紧凑的 M-bytecode（压缩率 9.8x vs zlib-JSON, 9.6x vs CBOR），在 MVM 上以 capability gating 和 step-limit 实现运行时有界执行（bounded execution）。在确定性解码模式（temperature=0）下，113 个确定性任务的 M-IR $\to$ M-bytecode 链路含 repair 全线通过率达 1.000，repair loop 的 success@1 为 0.982；在 96 个闭环任务上，通过 prompt 工程消除 retry/repeat 语义混淆后 task\_success\_rate 达 0.948。在非零温度采样模式下成功率显著下降（0.6 时 0.909±0.289，1.0 时 0.847±0.361），表明当前框架的可靠性依赖于确定性解码。有界执行实验验证了 UABR=1.000 的异常阻断能力。实验结果表明该框架在确定性控制与闭环语义子集上均具有可行性，但在采样模式下的鲁棒性仍需进一步研究。

\textbf{关键词}：大语言模型；硬件控制；中间表示；字节码；虚拟机；嵌入式系统
\end{abstract}

\section{引言}
\label{sec:intro}

\subsection{背景与动机}
\label{sec:background}

将大语言模型生成的控制程序部署到资源受限微控制器（MCU）上，是 AI 驱动硬件控制的核心落地挑战。已有工作证明 LLM 生成控制逻辑的可行性：Code-as-Policies~\cite{liang2023code} 使用 LLM 生成高层 Python policy 来控制机器人臂，RT-2~\cite{brohan2023rt2} 将视觉语言模型直接输出为机器人动作序列。然而，当目标设备从桌面级机器人变为 ESP8266 等资源受限 MCU 时，问题的约束条件发生了本质变化。考虑一个具体场景：用户对系统说"打开水泵并读取水位传感器"，系统需要将这条自然语言指令转化为可在 MCU 上安全执行的控制程序。这一过程至少面临四类约束：文本表示是否适合 LLM 稳定生成，编译后程序是否足够紧凑以便低带宽链路传输，设备端执行是否有界且可审计，以及当模型输出错误程序时系统能否阻断风险并回传结构化错误。

\subsection{现有方法的局限}
\label{sec:limitations}

现有方案难以同时满足上述约束。JSON 接口在 IoT 场景中广泛使用，易于生成且生态成熟，但文本冗余高，一条简单控制指令的 JSON 编码可达百字节量级，不利于带宽受限的 MCU 链路。Python 等高层代码表达力强，但端侧执行需要完整解释器，资源代价高，且缺乏形式化的执行边界约束。WASM/WAMR 等通用字节码运行时和 grammar-constrained decoding 等生成约束技术各自解决了问题的一个侧面——前者关注沙箱与可移植性，后者关注输出格式合法性——但均未将 LLM 的生成成本、编译紧凑性、端侧有界执行和闭环反馈作为一个统一问题处理。

\subsection{问题定义与目标}
\label{sec:problem}

本文探索能否设计一种既适合 LLM 生成、又能编译为紧凑字节码并在 MCU 上安全执行的控制表示。具体而言，系统需同时满足六项设计目标：\texttt{token-efficient}（生成紧凑）、\texttt{compact}（传输紧凑）、\texttt{deterministic-to-compile}（编译确定）、\texttt{statically-checkable}（静态可验证）、\texttt{bounded-to-execute}（执行有界）和 \texttt{auditable-on-fault}（故障可审计），详见 \S\ref{sec:goals}。

为此提出两层框架 M-IR + M-bytecode + MVM，将"生成友好"和"执行友好"分层优化。\textbf{核心思路}：将控制表示拆分为两层——M-IR（面向 LLM 的文本 DSL）负责生成友好，M-bytecode（面向 MCU 的紧凑字节码）负责执行友好，二者通过确定性 compiler 连接；生成与执行不再被强行揉进一个表示，而是各自独立优化。

注：M-IR 的输出紧凑性以更长的 system prompt 为代价，在 prompt 可缓存的场景下总成本优势才会显现（详见 \S\ref{sec:token_analysis}）。这一两层设计体现了一种更广泛的设计思路：\textbf{LLM 与受限环境交互的分层解耦}——上层面向模型的自然表达习惯（生成友好），下层面向执行环境的物理约束（执行友好），二者通过确定性编译连接。\S\ref{sec:problem_def} 给出形式化的问题定义与评价指标。

\subsection{关键挑战}
\label{sec:challenges}

实现上述目标面临以下技术挑战：

\begin{enumerate}
  \item \textbf{生成-执行鸿沟}：LLM 擅长结构化文本，但紧凑字节码对模型而言不可读、不可调试；同时，IR 到字节码的 lowering 必须确定且无歧义，任何 ambiguity 都会导致不可预测的端侧行为。这两层之间的张力——既要生成友好又要执行紧凑、既要灵活表达又要确定性编译——是框架设计的核心矛盾。
  \item \textbf{端侧有界执行}：AI 生成的程序本质上不可信，需要在 MCU 上通过 capability 约束、步数限制和结构验证确保执行不会越权或死循环。
  \item \textbf{闭环反馈与修复}：当执行失败时，系统需要将结构化错误信息回传给 LLM 以触发修复，而非仅返回不可解析的故障码。
\end{enumerate}

\subsection{解决方案概述}
\label{sec:overview}

本文采用两层设计以应对上述挑战：

\begin{enumerate}
  \item \textbf{M-IR}（面向挑战 1 的生成侧）：面向 LLM 输出的结构化中间表示，强调 token 效率、可编译性和可调试性，支持 capability 声明、读回验证与重试语义。
  \item \textbf{M-bytecode}（面向挑战 1 的执行侧）：由 compiler 确定性 lowering 生成的紧凑字节码，采用 varint 编码与栈式执行模型，体积显著低于 JSON/CBOR/MessagePack。
  \item \textbf{MVM 与静态验证器}（面向挑战 2）：在 MCU 上提供 capability gating、step-limit、结构验证与 fault audit，确保有界执行。
  \item \textbf{生成链路与 repair loop}（面向挑战 3）：打通 \texttt{LLM $\to$ M-IR $\to$ bytecode $\to$ verify $\to$ execute $\to$ feedback $\to$ repair} 的完整路径，支持结构化错误回传与自动修复。
\end{enumerate}

"生成友好"和"执行友好"不再被强行揉成一个层次，而是分别优化。以贯穿全文的 running example 为例：用户输入"打开水泵并读取水位传感器"，LLM 输出 6 行 M-IR，compiler 将其 lowering 为 12 字节 M-bytecode，在 ESP8266 上以 6 steps 完成执行（完整链路见 \S\ref{sec:pipeline_overall}）。

\subsection{目标场景}

本文面向一类离散或中低频控制任务：上位机或云端 AI 根据自然语言任务描述和设备白名单生成控制程序，端侧 MCU 接收该程序并执行，同时返回故障码、读回值和执行状态。

\end{enumerate}

\subsection{核心指标定义}
\label{sec:metric_defs}

本文使用以下核心指标，此处给出形式化定义以确保可复现性。

\textbf{压缩率}：设 $B_{\text{M}}(t)$ 和 $B_{\text{JSON}}(t)$ 分别为任务 $t$ 的 M-bytecode 与 JSON payload 字节数，则平均压缩率与加权压缩率定义为：

\begin{equation}
R_{\text{AVG}} = \frac{1}{|T|} \sum_{t \in T} \frac{B_{\text{JSON}}(t)}{B_{\text{M}}(t)}, \quad
R_{\text{WEIGHTED}} = \frac{\sum_{t \in T} B_{\text{JSON}}(t)}{\sum_{t \in T} B_{\text{M}}(t)}
\label{eq:compression}
\end{equation}

\textbf{任务成功率（Task Success Rate）}：设 $T$ 为任务集，$\mathbf{1}[\cdot]$ 为指示函数，则：

\begin{equation}
\text{TSR} = \frac{1}{|T|} \sum_{t \in T} \mathbf{1}[\text{task } t \text{ passes all stages}]
\label{eq:tsr}
\end{equation}

其 95\% Wilson 置信区间按标准二项比例方法计算。

\textbf{未授权访问阻断率（UABR）}：在安全实验中，设 $N_{\text{blocked}}$ 为被 capability gating 或 validator 阻断的未授权请求数，$N_{\text{total}}$ 为总未授权请求数，则 $\text{UABR} = N_{\text{blocked}} / N_{\text{total}}$。

\textbf{安全收敛率（SCR）}：在闭环实验中，经过 $k$ 轮 retry 后达到目标状态的任务比例。

\textbf{Repair success@k}：在 repair loop 中，第 $k$ 轮修复后首次成功的任务数占总任务数的比例。

\subsection{威胁模型}
\label{sec:threat}

\textbf{安全范围说明}：本文所称"安全"特指运行时有界执行（capability 约束、step-limit、结构验证、故障审计），不涵盖链路认证、加密传输、供应链安全或物理对抗等更广泛的安全议题。

本文默认：

\begin{enumerate}
  \item LLM 输出不可信；
  \item 外部输入可能被构造、篡改或重放；
  \item validator、执行器和基础 I/O 驱动是当前可信计算基；
  \item 链路认证、加密、供应链攻击与物理对抗不在本文验证范围内。
\end{enumerate}

本文所验证的"安全"边界仅限于：\textbf{运行时有界执行}，即通过 capability gating、step-limit、静态验证和 fault audit 确保 MCU 执行的程序不会越权、不会无限循环、不会产生未审计的故障。这与更广泛意义上的"系统安全"（涵盖认证、加密、防篡改等）有明确区分。

\section{相关工作}

\subsection{资源受限 VM 与字节码执行}
\label{sec:rw_vm}

TinyVM~\cite{tinyvm} 面向嵌入式 C 子集提供轻量字节码解释器，Darjeeling~\cite{darjeeling} 为 FPGA 上的软处理器设计了 Java 子集 VM，AtomVM~\cite{atomvm} 在 ESP32 上实现 Erlang/BEAM 子集。WASM/WAMR~\cite{wamr} 提供跨平台字节码沙箱，eBPF 在 Linux 内核中实现受限程序的有界执行。这些系统主要强调通用脚本执行、可移植性、沙箱隔离或轻量部署。

本文与它们的差异在于：本文把 \textbf{LLM 生成成本} 明确纳入设计目标，而不是只做"又一个轻量 VM"。M-IR 的语法设计以 LLM 的 token 效率为出发点，M-bytecode 的编码方案以传输体积为优化目标，这是现有嵌入式 VM 工作未涉及的维度。

从编译器 IR 设计的视角看，M-IR 的设计借鉴了经典 IR 的分层思想：类似 LLVM IR 在高层语言与机器码之间提供抽象层，或 WASM 文本格式（WAT）在可读性与紧凑二进制（WASM）之间建立桥梁~\cite{llvm_ir,spirv}。M-IR 的独特之处在于其"面向 LLM 生成"的设计约束——IR 设计通常以编译器分析和优化为目标，而 M-IR 的语句集（set/read/wait/retry 等）直接映射到控制任务的领域语义，牺牲了一般性以换取生成稳定性。关于 step-limit 和 capability gating 机制，本文的设计与实时 Java 安全子集（如 SCJ~\cite{scj}）和沙箱指令计数（如 Google Native Client~\cite{nacl}）共享"有界执行"的核心思想，但在 MCU 级实现上做了显著简化——MVM 以约 500 行 C++ 代码实现完整的栈机 + 22 种 fault 码 + 串口协议，无需操作系统支持。

值得进一步澄清的是：\textbf{为什么 WASM/WAMR 或 eBPF 不能简单适配以解决本文的问题？} WASM 的文本格式（WAT）使用 S-expression 语法，LLM 对其的熟悉度远低于类 C 赋值语句，且在 MCU 上运行完整的 WASM 解释器（即使是最小配置的 WAMR）需要数十 KB 的 RAM 开销，远超 MVM 的约 2KB 栈空间。eBPF 的指令集为内核级数据平面优化，缺少 MCU 硬件控制所需的 GPIO 读写、模拟量采样和毫秒级等待原语。此外，两者均不提供 M-IR 的 \texttt{require cap(...)} 设备能力声明和 \texttt{readback/retry} 闭环语义的原生支持——这些语义需要在应用层额外编码，破坏了端到端的紧凑性。

\subsection{LLM 代码生成与机器人控制}
\label{sec:rw_llm}

Code-as-Policies~\cite{liang2023code} 使用 LLM 将自然语言指令解析为 Python 代码片段，通过程序合成控制机器人臂执行抓取、搬运等任务。其优势在于 Python 的表达力和生态，但 Python 解释器的资源需求使其难以部署在 MCU 上。RT-2~\cite{brohan2023rt2} 将视觉语言模型直接微调为动作输出模型，实现了端到端的机器人控制，但其设计面向桌面级机器人平台，不涉及资源受限设备的执行约束。

本文关注更窄但更工程化的问题：如何生成适合 MCU 落地、可验证、可编译的控制表示。与 Code-as-Policies 相比，本文将执行层从 Python 下降到字节码；与 RT-2 相比，本文保留了显式的 IR 编译路径而非端到端映射。

\subsection{IoT 控制接口}
\label{sec:rw_iot}

JSON / text command 在 IoT 场景中广泛存在，MQTT、CoAP 等协议的 payload 通常采用 JSON 编码。其优点是简单与生态成熟；缺点是冗余高，一条简单控制指令的 JSON 编码可达百字节量级。CBOR 和 MessagePack 作为二进制替代方案，能减少约 30--50\% 的体积，但仍保留了通用序列化格式的结构开销。需说明的是，CBOR 和 MessagePack 为二进制序列化格式，其设计目标并非供 LLM 直接生成；让 LLM 输出二进制 payload 面临与 Direct Hex 基线（\S\ref{sec:exp_c}）相同的根本困难——模型无法稳定生成二进制编码，且错误无法被结构化解析与修复。因此，本文的比较聚焦于编译后体积（表~\ref{tab:encoding}），即 M-IR 经确定性 compiler 后的字节码与同等语义的 JSON/CBOR/MessagePack payload 的对比。现有方案均未同时优化生成成本、传输体积和端侧执行边界。

\subsection{本文定位}
\label{sec:rw_positioning}

本文位于三类工作交叉处：

\begin{enumerate}
  \item 借用 LLM 的生成能力；
  \item 借用嵌入式 VM 的有界执行思想；
  \item 借用 IoT 控制任务的设备语义与部署场景。
\end{enumerate}

\subsection{结构化生成与约束解码}
\label{sec:rw_structured}

近年来，结构化生成技术（如 JSON mode、grammar-constrained decoding~\cite{willard2023structured}）已被广泛用于强制 LLM 输出符合特定格式。这类方法的核心思路是在解码阶段施加语法约束，使输出必定满足预定义的 schema。

本文与这类工作的关键差异在于：约束解码解决的是\textbf{输出格式合法性}问题，而 M-IR 还额外解决了以下问题：

\begin{enumerate}
  \item \textbf{可编译性}：M-IR 不仅是合法文本，还能被确定性地 lowering 为紧凑字节码，这是 JSON Schema 约束无法保证的；
  \item \textbf{设备能力声明}：M-IR 的 \texttt{require cap(...)} 语法将设备能力约束嵌入表示层，而非依赖外部校验；
  \item \textbf{闭环语义集成}：\texttt{readback/retry} 等闭环操作被编译为字节码级控制流，而非留在文本层由外部解释；
  \item \textbf{端侧有界执行}：M-bytecode 通过 capability gating 和 step-limit 在 MCU 上提供运行时安全保障，这是输出格式约束完全无法覆盖的。
\end{enumerate}

因此，即使使用 grammar-constrained decoding 强制 LLM 输出合法 JSON，得到的仍然是需要端侧解析和执行的文本 payload，不具备 M-IR $\to$ M-bytecode $\to$ MVM 链路的紧凑性和有界执行优势。约束解码可以作为 M-IR 生成的补充手段（例如用 grammar 约束提高 M-IR 的首轮合法率），但不能替代 M-IR 的设计本身。

\section{M-IR 与 M-bytecode 设计}
\label{sec:mir}

\subsection{两层表示}
\label{sec:two_layer}

如 \S\ref{sec:overview} 所述，本文采用两层表示：

\begin{enumerate}
  \item \textbf{M-IR}：面向 LLM 输出的结构化控制表示；
  \item \textbf{M-bytecode}：由 compiler 生成、由 MVM 执行的紧凑字节码。
\end{enumerate}

M-IR 负责稳定生成与可调试，M-bytecode 负责紧凑传输与低开销执行。

\subsection{M-IR 设计目标}
\label{sec:mir_goals}

M-IR 目标是：

\begin{enumerate}
  \item 相比 JSON 更紧凑；
  \item 相比裸 hex bytecode 更易生成和修复；
  \item 可被确定性 lowering 到 M-bytecode；
  \item 能显式表达 capability、等待、读回与重试语义。
\end{enumerate}

\textbf{设计选择论证}：M-IR 采用文本 DSL 而非 S-expression 或栈式表示，主要基于以下考量：(1) LLM 在预训练中接触大量类 C 语法的代码，`set relay1 = 1` 等赋值式语句比 `(set relay1 1)` 的 S-expression 或 `1 relay1 SET` 的栈式表示更接近模型的自然分布，有利于稳定生成；(2) 文本 DSL 便于人类调试和 prompt 工程迭代，错误定位直观；(3) 每行一条语句的格式使 compiler 能按行规范化，降低解析复杂度。\texttt{wait} 以 ms 为单位（而非 cycle 或 tick）是为了与硬件控制中常见的毫秒级时序语义保持一致，降低 LLM 理解成本；\texttt{retry} 当前使用 \texttt{times}（固定次数）而非 \texttt{until}（条件谓词），是在实现复杂度与表达力之间的工程权衡——条件 \texttt{retry\_until} 需要 compiler 支持表达式求值和布尔栈操作，已列入后续演进计划（\S\ref{sec:future}）。当前语句子集（set/read/wait/readback/retry/repeat/halt）覆盖了 \S\ref{sec:problem_def} 问题定义中的确定性控制、时序序列、闭环保持与错误恢复模式，但尚未包括条件分支（\texttt{if/else}）的原生语法支持——当前条件分支通过 \texttt{retry} 的隐式 readback 比较间接实现，完整的条件控制流语法留待后续工作。

本文固定采用 textual DSL 作为 M-IR v0.1 的表层格式。M-IR 的语法可由以下 EBNF 文法定义：

\begin{lstlisting}[style=mirstyle,caption={M-IR EBNF 文法},label={lst:grammar}]
<program>    ::= <stmt>+
<stmt>       ::= <cap_decl> | <set_stmt> | <read_stmt>
               | <wait_stmt> | <halt_stmt>
               | <readback_stmt> | <retry_stmt>
<cap_decl>   ::= "require" "cap" "(" <device> ")"
<set_stmt>   ::= "set" <device> "=" <bit>
<read_stmt>  ::= "read" <device>
<wait_stmt>  ::= "wait" <number> "ms"
<halt_stmt>  ::= "halt"
<readback_stmt> ::= "readback" <device> "expect" <bit>
<retry_stmt> ::= "retry" <number> "times" "{" <stmt>+ "}"
<device>     ::= "water_sensor" | "temperature_sensor"
               | "humidity_sensor" | "relay1" | "relay2"
<bit>        ::= "0" | "1"
<number>     ::= [1-9][0-9]*
\end{lstlisting}

当前语法子集支持七类语句（以 running example 中的设备为例）：

\begin{enumerate}
  \item \texttt{require cap($\langle$device$\rangle$)}：声明设备能力
  \item \texttt{set $\langle$device$\rangle$ = $\langle$0|1$\rangle$}：控制设备开关
  \item \texttt{read $\langle$device$\rangle$}：读取传感器值
  \item \texttt{wait $\langle$ms$\rangle$ms}：延时等待
  \item \texttt{halt}：终止执行
  \item \texttt{readback $\langle$device$\rangle$ expect $\langle$value$\rangle$}：读回验证
  \item \texttt{retry $\langle$n$\rangle$ times \{ ... \}}：重试控制流
\end{enumerate}

\textbf{Running Example}（贯穿全文）：以"打开水泵并读取水位传感器"为例，LLM 生成的 M-IR 如下：

\begin{lstlisting}[style=mirstyle,caption={Running Example：LLM 生成的 M-IR},label={lst:example}]
require cap(relay1)
require cap(water_sensor)
set relay1 = 1
wait 500ms
read water_sensor
halt
\end{lstlisting}

\section{MVM 与静态验证器}
\label{sec:mvm}

\subsection{MVM 架构}
\label{sec:mvm_arch}

MVM 在 MCU 上解释执行 M-bytecode。当前 ESP8266 实验实现使用如下状态：

\begin{equation}
\texttt{pc / stack / locals / ret-stack / call-depth / steps / fault}
\label{eq:mvm_state}
\end{equation}

并提供统一响应：

\begin{enumerate}
  \item OK：\texttt{[result varint][steps varint][0x01]}
  \item FAULT：\texttt{[fault varint][pc varint][0x00]}
\end{enumerate}

\subsection{静态验证器}
\label{sec:validator}

主机侧 static verifier（静态验证器，下文统一称 static verifier）已实现：

\begin{enumerate}
  \item opcode 合法性；
  \item varint 完整性；
  \item block 结构匹配；
  \item locals 访问合法性；
  \item 部分可达性与结构一致性检查。
\end{enumerate}

\subsection{运行时保护机制（有界执行）}
\label{sec:safety}

MVM 当前通过以下机制提供有界执行：

\begin{enumerate}
  \item capability gating
  \item step-limit
  \item call-depth-limit
  \item stack / locals / pc 边界检查
  \item violation $\to$ fault
\end{enumerate}

每条 M-bytecode 指令消耗固定步数（见附录~\ref{app:inst_table} 指令集表），MVM 每执行一条指令步数计数器 +1，达到 step-limit 时触发 fault。因此，在任何输入下，MVM 执行步数 $\leq$ step-limit，保证终止。

\subsection{闭环核验接口}
\label{sec:closed_loop}

当前系统支持：

\begin{equation}
\texttt{set $\to$ readback $\to$ compare $\to$ retry}
\label{eq:closed_loop}
\end{equation}

这里的重点不是声称"闭环语义内建一定更优"，而是提供统一的观测与修复接口。

\section{LLM $\to$ M-IR $\to$ Bytecode 生成链路}
\label{sec:pipeline}

\subsection{总体链路}
\label{sec:pipeline_overall}

如 \S\ref{sec:two_layer} 所述的两层架构，本文采用如下具体链路（完整设计见第\ref{sec:mir}、\ref{sec:mvm}章）：

\begin{equation}
\text{Natural Language} \to \mir{} \to \text{Compiler} \to \text{Verifier} \to \mvm{}/\text{Simulator} \to \text{Feedback} \to \text{Repair}
\label{eq:pipeline}
\end{equation}

以贯穿全文的 running example（"打开水泵并读取水位传感器"）为例，完整链路如下：

\begin{enumerate}
  \item \textbf{Natural Language}：用户输入"打开水泵，等待 500ms，读取水位传感器"
  \item \textbf{LLM $\to$ M-IR}：模型输出上述 M-IR 文本
  \item \textbf{Compiler}：将 M-IR lowering 为 M-bytecode（\texttt{GTWAY 5; GTWAY 1; LIT 1; IOW 5; WAIT 500; IOR 1; HALT}，共 12 字节）
  \item \textbf{Verifier}：静态检查通过（opcode 合法、varint 完整、capability 已声明）
  \item \textbf{MVM 执行}：在 ESP8266 上解释执行，完成设备控制与传感器读取
  \item \textbf{Feedback}：返回 \texttt{[result=0x01, steps=6, OK]}
\end{enumerate}

若任何阶段失败，结构化错误信息回传 LLM 触发 repair loop。

\subsection{为什么不直接生成裸字节码}
\label{sec:why_not_raw}

直接生成裸 bytecode 的主要问题是：

\begin{enumerate}
  \item 文本层不可读；
  \item 错误难以定位；
  \item 无法稳定回传结构化修复信息。
\end{enumerate}

因此本文明确采用分层生成，而不要求 LLM 直接输出 hex payload。

\subsection{当前 compiler 状态}
\label{sec:compiler_status}

当前最小 compiler 已支持：

\begin{enumerate}
  \item \texttt{require cap(...)}
  \item \texttt{set}
  \item \texttt{read}
  \item \texttt{wait}
  \item \texttt{halt}
\end{enumerate}

同时，\texttt{readback/retry} 已进入统一编译链：前者 lowering 为 \texttt{IOR + LIT + EQ}，后者 lowering 为显式重试控制流。\texttt{compile\_to\_plan(...)} 作为调试视图与宿主侧编排接口存在，但不再是闭环语义的唯一载体。

\subsection{反馈与修复}
\label{sec:feedback}

当前反馈 schema 已固定为：

\begin{enumerate}
  \item \texttt{stage}
  \item \texttt{error\_code}
  \item \texttt{message}
  \item \texttt{hint}
\end{enumerate}

repair loop 据此对 parse error、capability error、task mismatch 等错误进行结构化修复。

\subsection{对三个研究问题的回答}
\label{sec:rq_answers}

\subsubsection{RQ1：token 效率与字节码紧凑性}

\textbf{RQ1：部分回答。}

在编译后体积方面，M-bytecode（平均 10.92B）相对公平基线——zlib-JSON（106.7B, 9.8$\times$）、CBOR（97.77B, 9.0$\times$）、MessagePack（97.42B, 8.9$\times$）——具有约 9--10 倍压缩优势（注：LLM 原始 JSON 为 285.90B / 26$\times$，属不公平对比，因未经编译/压缩处理；公平对比应以 zlib-JSON 或 CBOR 为参照）。该优势源于指令级紧凑设计而非文本格式冗余。在 RTT 方面，观测到 M-bytecode 路径在所有 I/O 任务上 RTT 中位数均低于 JSON 路径（差距约 2ms），该差异方向一致但工程意义有限（详见 §8.1 讨论）。

在 token 成本方面，M-IR 存在一个重要的 tradeoff：其输出 token 显著低于 JSON（40.00 vs 77.54，在 52 个三方语义等价任务上），但 system prompt 更长（161.15 vs 83.15 tokens），导致单次调用的总生成成本偏高（201.15 vs 160.69）。然而，在 system prompt 可缓存的场景下（如 Anthropic Prompt Caching），M-IR 的增量成本仅为 40.00 output tokens——是 JSON 的 51.6\%；在批量生成或多轮交互中，system prompt 被复用后 M-IR 的总成本优势随调用次数累积（详见 \S\ref{sec:token_analysis}）。

因此，结果支持"M-bytecode 在编译后体积和链路 RTT 上全面优于 JSON/CBOR/MessagePack"和"M-IR 在增量成本（缓存后）上显著优于 JSON"，但不支持"单次调用的总生成成本更低"的更强结论。

\subsubsection{RQ2：M-IR 生成链路可靠性}

\textbf{RQ2：已获得较强实证支撑。}
113 个确定性任务覆盖 7 种控制模式（开关控制、模拟量读取、条件等待、时序序列、组合任务、闭环保持、错误恢复）。在此集合上，使用本地 \texttt{llama3.1:8b}（temperature=0.0）生成 M-IR 后，全线通过率达到 \texttt{1.000}。repair loop 的 \texttt{success@1} 为 \texttt{111/113 = 0.982}，第二轮修复剩余 \texttt{2} 个样本。temperature 扫描实验进一步表明，该通过率部分依赖确定性解码：\texttt{temperature=0.6} 时 \texttt{task\_success\_rate} 降至 \texttt{0.909 $\pm$ 0.289}，\texttt{temperature=1.0} 时降至 \texttt{0.847 $\pm$ 0.361}。值得关注的是，temperature=0.6 时标准差 0.289 意味着单次运行的通过率在 0.620 到 1.000 之间剧烈波动，temperature=1.0 时更甚（std=0.361）。这种不稳定性意味着：若不使用确定性解码（temperature=0），M-IR 链路在生产部署中的可靠性无法保证，未来需探索 grammar-constrained decoding + M-IR 的结合方案。主要失败模式为 M-IR 格式偏差（\texttt{ir\_valid\_rate} 下降），说明 M-IR 语法约束在采样模式下仍有一定鲁棒性，但需配合 grammar-constrained decoding 或 repair loop 保证高可靠性。

同时，多模型对比表明该链路并非只对单一模型成立。\texttt{qwen3:8b} 在同一 113 任务集上的 \texttt{task\_success\_rate} 为 \texttt{0.903}（95\% CI: [0.835, 0.946]），主要错误为 \texttt{TASK\_SIGNATURE\_MISMATCH} 与 \texttt{MIR\_PARSE\_ERROR}；\texttt{deepseek-r1:8b} 则达到 \texttt{1.000}（95\% CI: [0.968, 1.000]）。三个 8B 模型中两个满分通过，说明 M-IR 链路对主流小模型具有较好兼容性，但模型选择仍影响最终成功率。

进一步地，本文将生成实验扩展到控制流构造。在 15 个覆盖 \texttt{if/else} 条件分支、\texttt{repeat} 循环及其嵌套组合的新任务上，\texttt{llama3.1:8b} 的 \texttt{task\_success\_rate} 为 \texttt{0.867}（13/15），两个失败样本分别为语义偏差和不支持语法的使用，均为 LLM 生成偏差而非编译器缺陷。这表明 M-IR 的结构化语法约束对控制流生成同样有效。

在闭环语义方面，早期在 16 个任务的探索性子集上进行了预实验（\texttt{task\_success\_rate = 0.938}，95\% CI: [0.717, 0.997]），但由于样本量小、置信区间宽度达 0.28，该结果仅作为方向性参考，不做为主要结论依据。将闭环任务集扩展至 96 个后，本文通过 prompt 工程消融实验明确了 \texttt{retry$\to$repeat} 混淆的根因：使用未包含语义区分说明的基础 prompt 时，\texttt{task\_success\_rate} 仅为 0.542，41/96 个失败为 \texttt{TASK\_SIGNATURE\_MISMATCH}（其中 34 例为 \texttt{retry$\to$repeat} 混淆）；在 prompt 中增加 \texttt{retry} 与 \texttt{repeat} 的语义区别说明及两条 few-shot 示例后，\texttt{task\_success\_rate} 跃升至 0.948，仅余 5 例 \texttt{MIR\_PARSE\_ERROR}（偶发语法偏差）。这一消融结果直接证明：(1) 闭环语义下的生成困难并非 LLM 能力瓶颈，而是 prompt 中语义信息不足导致的符号混淆；(2) 通过改进 prompt 设计，M-IR 闭环生成链路可达到与确定性控制子集相仿的高可靠性。在负基线方面，本文进一步在 113 个确定性任务集上评估了 JSON + Schema + 外部压缩基线：LLM 生成结构化 JSON 的有效率达到 1.000，任务成功率 0.965，但经 zlib level 9 压缩后体积仍为 M-bytecode 的 9.8 倍（106.7 vs 10.9 bytes），且压缩后的 JSON 缺少确定性编译、静态验证和有界执行保证。这一对比从"生成-执行鸿沟"的角度进一步支撑了分层生成路径的必要性：仅有生成友好性（JSON 同样可达 1.000 生成有效率）不足以保证执行安全性，还需编译器的确定性桥接与后端的受限执行机制。

\subsection{Token 成本与 Context Length 分析}
\label{sec:token_analysis}

表~\ref{tab:token} 显示 M-IR 的总 token 成本（201.15）高于 JSON（160.69），主要源于更长的 system prompt（M-IR 平均 161.15 tokens vs JSON 平均 83.15 tokens）。本节进一步分析 system prompt 被缓存或复用场景下的成本变化。

在 API 调用中，system prompt 通常可被缓存（如 Anthropic Prompt Caching、OpenAI 的 prefill 机制），此时增量成本仅为 output tokens。附录 C 表 14 给出缓存场景下的增量成本对比。需注意 token 对比实验基于 52 个"三方语义等价"任务子集——仅纳入 M-IR、JSON、Python 三种格式均生成语义正确程序的样本，以排除语义错误对 token 计数的干扰，确保对比仅反映格式本身的表达效率差异。

当 system prompt 被缓存后，M-IR 的增量输出 token（40.00）仅为 JSON（77.54）的 51.6\%，优势显著。PYTHON 仍以最低输出 token（17.19）领先，但 PYTHON 在语义通过率上最低（0.690，表~\ref{tab:semantic}），综合可行性不如 M-IR。

进一步地，考虑不同对话轮次下 system prompt 被复用的累积效应。假设同一 system prompt 被复用 $k$ 次，则平均每次调用的总成本为 $\text{prompt\_tokens} / k + \text{output\_tokens}$。当 $k \geq 2$ 时，M-IR 的总成本（$201.15/k$）开始低于 JSON（$160.69/k$）的等效值，因为 M-IR 的输出 token 优势在累积调用中被放大。因此，M-IR 的成本优势在多轮交互或批量生成场景中更为显著。

\section{讨论与局限}
\label{sec:discussion}

\subsection{当前最重要的局限}
\label{sec:limitations_detail}

\begin{enumerate}
  \item 当前实机验证的是控制子集，而不是完整 M 语言；
  \item 确定性子集（113 任务）与控制流子集（15 任务）的生成实验已分别验证。闭环任务集已扩展至 96 个任务（覆盖 12 个子类别）。通过 prompt 工程消融实验，\texttt{task\_success\_rate} 从基础 prompt 的 0.542 提升至优化 prompt 的 0.948，确认 \texttt{retry$\to$repeat} 混淆可通过 prompt 设计消除。回到 \S\ref{sec:challenges} 的核心矛盾——生成-执行鸿沟：两层架构的分层设计使得 prompt 层的改进（生成友好）可独立于执行层进行，闭环成功率的显著提升正是这一架构优势的体现；
  \item \texttt{readback/retry} 已经完成最小 bytecode lowering，但当前尚未完成板级统一 runner 验证；
  \item token 对比虽然已扩大到 52 个三方等价任务，但仍然主要局限于确定性任务族；
  \item 当前结果主要基于 ESP8266 与 ESP32 实机验证。STM32 等更多平台的实机验证尚未开展，MVM 在异构 ISA（如 ARM Cortex-M）上的移植性尚未被验证；
  \item 当前确定性生成（temperature=0）下的 1.000 通过率无法外推至采样模式（temperature$>$0），后者在实践中更常见。在 temperature=1.0 时，成功率标准差高达 0.361，表明单次运行的可靠性不可控；
  \item 当前强结论（1.000 通过率）主要基于 llama3.1:8b。qwen3:8b 同任务集成功率为 0.903（95\% CI: [0.835, 0.946]），表明 M-IR 链路的可靠性存在模型间差异。此外，本文未在更大参数模型（如 70B+）或商业闭源模型上验证，尚不清楚模型规模对 M-IR 生成稳定性的影响方向；
  \item 强实时控制与 AI-AI communication 尚未实证；
  \item 本文假设 LLM 输出不可信，但未讨论对抗性场景（如通过提示注入构造绕过 capability 检查的 M-IR 语法变体）。M-IR 的语法约束（EBNF 文法 + validator）在第一层防御中可拦截格式偏差，但未针对恶意构造的对抗性输入进行专门设计，相关防御（如对抗性训练、语法强化）仍有待探索。
\end{enumerate}

\subsection{与 LLM Agent 生态的关联}
\label{sec:agent_ecosystem}

本文的工作可以放在更大的 LLM Agent 生态中理解。

\textbf{与 tool-use / function-calling 的关系}：当前主流 LLM Agent 框架（如 LangChain、AutoGPT）通过 function-calling 让模型调用外部工具。M-IR 可视为一种面向嵌入式设备的结构化 tool definition：\texttt{require cap(...)} 对应工具的权限声明，\texttt{set/read/wait} 对应工具的参数与返回值，\texttt{readback/retry} 对应工具的闭环验证语义。与 JSON Schema 定义的 tool 相比，M-IR 的优势在于它不仅是描述，还能被确定性编译为可执行字节码，从而将"工具定义"与"工具执行"统一在一条编译链中。具体而言，M-IR 可通过以下路径与现有 Agent 框架集成：(1) 将 M-IR 语法注册为 LangChain 的 custom tool definition format，编译步骤封装为 tool 的 post-processing hook；(2) 在 AutoGPT 等自主 Agent 中，将 MVM 串口反馈（fault code + 审计信息）作为 tool execution result 回传给 LLM，实现闭环修复。

\textbf{与代码生成安全的关系}：LLM 生成代码的安全性（code sandboxing）是当前 Agent 安全研究的热点。主流方案通过 Docker 容器或 gVisor 沙箱隔离 Python/JS 代码的执行。本文的 MVM 提供了一种更轻量的替代：通过 capability gating 和 step-limit 在 MCU 级别实现有界执行，无需操作系统级隔离。这对于资源受限的边缘设备（无法运行容器）尤其有价值。

\textbf{与 MCU 生态中 "LLM-as-compiler" 趋势的关系}：随着 Edge AI 的发展，越来越多的工作尝试将 LLM 部署在 MCU 侧或用于生成 MCU 代码。本文的 M-IR $\to$ M-bytecode 路径为这一趋势提供了一个参考架构：LLM 负责语义生成，编译器负责紧凑化与验证，VM 负责有界执行。这种分层解耦的设计使得各层可以独立优化，而不必要求 LLM 直接理解硬件约束。

\subsection{后续方向}
\label{sec:future}

\begin{enumerate}
  \item 针对生成-执行鸿沟在闭环语义下的持续挑战，优先改进 M-IR 语法设计（如将 \texttt{retry} 改为 \texttt{retry\_until} 以提升与 \texttt{repeat} 的区分度），并在更大闭环任务集上系统验证 \texttt{readback/retry} lowering 后的生成稳定性；
  \item 在更大任务集上深化 JSON + Schema 基线分析（如扩展到闭环语义、增加 Python 结构化 schema 变体），形成 M-IR / JSON / Python 的统一对比矩阵；
  \item 增加闭环任务集并推进到统一板级 runner；
  \item 做统一板级 runner，把 \texttt{simulate} 推进到 \texttt{board}；
  \item 扩展更多模型横比；
  \item 将 MVM 移植至 STM32 等 ARM Cortex-M 平台，验证异构 ISA 上的字节码兼容性；
\end{enumerate}

\label{sec:conclusion}

\textbf{Artifact Availability.} 本文所有实验代码、数据集、MVM 固件及编译器源码均已开源，仓库地址：\url{https://github.com/MYJ-GOD/M-IR-MVM}。数据集包括 \texttt{tasks\_v2.jsonl}（113 确定性任务）、\texttt{tasks\_v3\_expanded.jsonl}（96 闭环任务）、\texttt{tasks\_v4.jsonl}（15 控制流任务），均已标注 ground truth。固件支持 ESP8266 与 ESP32 Arduino IDE 直接烧录，跨平台移植仅需修改引脚映射。

本文的核心理论贡献在于提出并验证了"生成友好—执行友好"分层解耦的表示设计原则：将 LLM 输出的语义表示与 MCU 执行的物理编码分离，通过确定性编译桥接，使两者可独立优化而不相互掣肘。这一原则的具体实例化——M-IR + M-bytecode + MVM——在 113 个确定性任务和 96 个闭环任务上验证了其可行性。

实验结果表明：(1) M-bytecode 相对公平基线（zlib-JSON）具有 9.8$\times$ 的压缩优势，且附带确定性编译、静态验证和有界执行保证——这些是通用压缩无法提供的；(2) 在确定性解码模式下，M-IR 生成链路达到 1.000 全线通过率，repair loop 可在两轮内修复全部残余失败；(3) 在非零温度采样模式下，可靠性显著下降（0.6 时 0.909±0.289，1.0 时 0.847±0.361），表明当前框架的鲁棒性依赖确定性解码；(4) capability gating、validator 和 step-limit 各自构成独立的安全层，多层防御的 UABR 达 1.000；(5) 闭环语义的生成瓶颈（retry/repeat 混淆）可通过 prompt 工程系统性消除，成功率从 0.542 提升至 0.948。

本文的工作尚限于确定性子集和单一 MCU 架构（ESP8266/ESP32），采样模式下的可靠性、更大参数模型和异构平台（STM32 ARM Cortex-M）的验证是下一阶段的关键方向。分层解耦设计原则是否能推广至其他"LLM + 受限环境"场景（如 FPGA 配置生成、实时操作系统调度策略合成），值得进一步探索。

每条指令消耗固定 1 步。在任何输入下，MVM 从 \texttt{pc}=0 开始，每执行一条指令步数计数器 +1。设 step-limit 为 $L$，程序指令数为 $n$，则实际执行步数 $\leq \min(n, L)$。若程序包含由 \texttt{JZ/JMP} 构成的循环，$\texttt{LIT/DUP/JZ/JMP/SUB/DRP}$ 序列每轮迭代步数恒定，因此循环在有限步内终止或触发 step-limit fault。由此保证 MVM 在任何输入下均在有界步数内终止。

\section{附录 E：三种格式的 System Prompt 全文}
\label{app:prompts}

为便于评审人验证实验公平性，以下给出 M-IR、JSON、Python 三种格式的完整 system prompt。三种 prompt 均已尽力优化，包含等价的设备定义、任务描述和输出约束。

\subsection{M-IR System Prompt}

\begin{lstlisting}[style=mirstyle,caption={M-IR 生成 System Prompt},label={lst:prompt_mir}]
You generate M-IR programs for hardware control tasks.
Follow these rules exactly:
1. Output only M-IR text.
2. Do not output markdown fences.
3. Use only devices from the allow-list.
4. Every accessed device must first appear in require cap(...).
5. Use only these statements: require cap(...), set <d>=0|1, read <d>, wait <ms>ms, halt.
6. Do not use readback or retry in v0.1.
7. Only relay1/relay2 may appear on the left side of set.
8. The program must start with `task <task_id> {` and end with `}`.
9. End every task with halt.
10. `halt` must be the last statement inside the task block.
11. Do not add extra actions not explicitly required by the instruction.
12. If the instruction says "wait then read", the program must be exactly wait->read->halt after capabilities.
\end{lstlisting}

\subsection{JSON System Prompt}

\begin{lstlisting}[style=mirstyle,caption={JSON 生成 System Prompt},label={lst:prompt_json}]
You generate JSON control programs for hardware control tasks.
Output only JSON. No explanation, no markdown fences.
The JSON object must have exactly three fields: "task" (string), "cap" (array of device names), "body" (array of operations).
Allowed operations (each with an "op" field):
  {"op":"set","dev":"<d>","to":0|1} — set relay/actuator state
  {"op":"read","dev":"<d>"} — read sensor/relay value
  {"op":"wait","ms":<int>} — wait milliseconds
  {"op":"readback","dev":"<d>","expect":0|1} — verify device state (must be last in retry body)
  {"op":"retry","times":<int>,"do":[<ops>]} — retry body up to N times, body must end with readback
  {"op":"repeat","times":<int>,"do":[<ops>]} — repeat body exactly N times unconditionally
  {"op":"halt"} — stop execution, must be last in body
Rules: declare all used devices in "cap"; body must end with halt; retry body must end with readback.
IMPORTANT retry vs repeat: retry for "try/verify/ensure/confirm/read back" tasks; repeat for "repeat/loop/toggle/do N times" tasks.
\end{lstlisting}

\subsection{Python System Prompt}

\begin{lstlisting}[style=mirstyle,caption={Python 生成 System Prompt},label={lst:prompt_python}]
You generate compact Python-like control snippets for hardware control tasks.
Output only Python code.
Use only direct imperative control statements that fully represent the task.
\end{lstlisting}
