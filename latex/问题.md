一、方法学与实验设计问题（最严重）
基线对比不公平且具有误导性

表3中对比的是“M-bytecode (golden) 平均10.92字节”与“JSON (LLM原始输出) 平均285.90字节”，计算出的26.18倍压缩比是不公平的。因为M-bytecode是编译后的紧凑二进制，而JSON是LLM直接生成的文本。公平的对比应该是：M-IR → compiler → M-bytecode vs JSON → 通用压缩（如zlib）或二进制序列化（CBOR）。作者虽在第15页补充了JSON+zlib基线（9.8倍），但未更新主表3，导致正文核心对比数据仍是误导性的。

核心结论仅在极端条件（temperature=0.0）下成立，但被普遍化

全文反复宣称“全线通过率1.000”“task_success_rate=1.000”，但该结果仅在使用确定性解码（temperature=0.0） 时获得。在更实用的temperature=0.6和1.0下，成功率分别降至0.909±0.289和0.847±0.361，且标准差巨大（0.289~0.361），意味着单次运行成功率可能在0.62~1.00之间剧烈波动。摘要、引言和结论均未明确限定“在确定性解码模式下”，构成严重误导。

任务集为自建且未公开验证，存在过拟合风险

所有113个确定性任务和96个闭环任务均由作者自行构建，未引用任何公开benchmark或社区公认的任务集。审稿人无法判断任务是否偏向M-IR语法（例如刻意避免需要复杂控制流的场景）。缺乏与现有数据集（如SayCan、RoboTHOR子集）的交叉验证。

Python基线的比较不合理

表10显示Python语义通过率仅0.690，但Python代码根本无法在ESP8266上安全执行（需要完整解释器）。将Python作为“效率基线”属于苹果与橘子对比。Python更适合作为“表达力上限”参考，而非公平的效率或可靠性基线。

RTT优势的工程意义被夸大

作者展示M路径比JSON路径快约2ms，但自己承认“对于中低频IoT控制任务，RTT的绝对差异工程意义有限”。既然如此，此结果不应作为RQ1的主要支撑证据。建议删除或移至附录。

二、结果呈现与数据分析问题
选择性报告结果

反复强调1.000通过率，但对temperature>0时的严重不稳定性轻描淡写（仅在第19页提及标准差和CI宽度）。论文的摘要、图形摘要和结论中完全未提及此重要局限。

压缩率计算方式有误（表3应修正或移除）

如第1条所述，表3中的26倍对比不可接受。作者在第15页自行给出了更公平的JSON+zlib基线（9.8倍），这恰恰推翻了表3的结论，但表3却保留未改。这属于严重的内部矛盾。

闭环子集上的置信区间过宽，结论过早

早期16个任务闭环子集的task_success_rate=0.938，但95% CI为[0.717, 0.997]，宽度达0.28。作者仍将其作为“方向性参考”写入主文，但未在摘要中明确标注该结果不稳健。虽然后续扩展到96任务，但早期结论的呈现方式易误导读者认为闭环语义已完全解决。

三、论述与结论问题
“安全”一词严重过载，未与系统安全区分

全文反复使用“安全执行”“安全机制”，但实际仅验证了有界执行（capability gating、step-limit、fault audit）。未涉及加密、认证、完整性校验、防重放、供应链安全等系统安全核心议题。威胁模型（§2.6）虽然做了限定，但正文其他部分（如标题、摘要、图1标注、§5.3标题）仍使用“安全”一词，极易误导读者。

对“分层解耦范式”的理论贡献表述模糊

作者声称提出了一种“广泛范式”，但未给出可迁移的设计原则或形式化框架。其他研究者如何将此范式应用于“LLM + 其他受限环境”（如FPGA、实时操作系统）？论文缺乏一般性指导，仅停留在具体实现层面。

结论与证据脱节

结论声称“该范式在确定性控制子集上具有可行性”，但标题（“大语言模型生成的控制器…M-IR+M-bytecode+MVM两层框架”）和摘要并未强调“确定性子集”，普通读者会误以为适用于所有LLM生成场景。

四、结构与可读性问题
篇幅过长，大量重复内容

第4章、第5章、第6章多次重复描述M-IR语法、lowering关系、running example。例如表4（lowering关系）在§4.6和§4.4附近重复出现。全文可精简30%。

术语混杂，缺少集中定义

出现M-IR、M-bytecode、MVM、static verifier、compiler、repair loop、capability gating、step-limit、fault audit、SAFE分流、UABR、SCR等十余个专有名词，没有词汇表或首次出现时的集中解释。读者需要前后翻阅才能理解。

附录D关键内容缺失

附录D承诺给出M-IR、JSON、Python三种system prompt的全文，但实际写“待补充”。在提交评审时这是不可接受的，审稿人无法验证prompt工程是否公平（例如M-IR的prompt是否比JSON的prompt更长、更详细，导致不公平优势）。

五、学术规范与严谨性问题
未引用或讨论对抗性威胁相关文献

本文假设“LLM输出不可信”，但未讨论如何防御恶意构造的M-IR（例如通过提示注入生成绕过capability检查的语法变体）。相关文献（如针对LLM代码生成的对抗性攻击）完全缺失。

对已有相关工作的差异化论述不足

作者区分了与WASM/WAMR、eBPF、grammar-constrained decoding的差异，但未能有力论证“为什么这些技术不能通过简单适配解决本文问题”。例如，可否用约束解码直接生成紧凑的CBOR payload？为什么不行？现有论述停留在“它们没有同时优化生成成本”，这不够深刻。

跨平台验证的声明过于简化

作者称ESP32移植“仅改动3行引脚定义”，表明MVM核心与平台解耦。但未提供任何性能对比（如ESP32上的执行步数/时延是否与ESP8266一致），也未验证不同字节序（ESP8266小端，ESP32小端虽相同但未说明）或内存模型差异。声明缺乏数据支撑。

六、具体数据与表格问题
表1（研究问题与评价指标映射）过于抽象：RQ1对应的“token数、bytecode长度、RTT、执行时延”中“执行时延”在后续实验中并未系统给出（仅有部分RTT数据）。指标与实验不完全对应。

表5（temperature扫描）呈现方式不完整：只给出了均值和标准差，未给出每次运行的原始成功率，也未进行统计显著性检验（如ANOVA或Kruskal-Wallis）。方差巨大时，仅报告均值意义有限。

表9与表10的样本不一致：表9的token对比基于52个“三方语义等价任务”，而表10的语义通过率基于113任务。为什么是52个？筛选标准未说明，存在数据裁剪嫌疑。

七、表述与措辞问题
“token-efficient”定义模糊：§1.7将token-efficient解释为“生成紧凑（LLM输出token少）”，但实际M-IR的输出token确实少于JSON，然而总token（system prompt + output）却高于JSON。作者在§8.9中承认了这一点，但§1.7的定义未反映这个trade-off，属于定义不完整。

摘要中的数字表述不统一：“压缩率4.4x vs JSON，9x vs CBOR”与后续正文第15页的“9.8x vs zlib-JSON”不一致。摘要应使用最终公平对比的数据。

多处使用绝对值比较而未给出统计检验：例如“M路径RTT始终低于JSON路径（差距约2ms）”，未给出p值或置信区间。虽然差距明显，但缺少统计显著性声明。

八、缺失内容
缺少与grammar-constrained decoding的直接结合实验：作者在§3.5讨论了该技术，在§9.1承认temperature>0时稳定性差，但没有进行任何结合grammar约束的实验来验证能否提高采样模式下的成功率。这属于明显的“未来工作”被伪装成“已讨论”。

缺少多轮交互累积成本的实验验证：§8.9理论上分析了system prompt缓存后M-IR的增量成本优势，但没有实际测量在对话历史变长、system prompt被截断或重新发送时的真实token消耗。理论分析不足以支撑结论。
