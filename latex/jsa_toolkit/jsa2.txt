A B S T R A C T
GPU Trusted Execution Environments (GPU-TEEs) protect GPU-accelerated AI applications from untrusted 
system software. Existing GPU-TEEs can be categorized into Type-I and Type-II designs. Type-I GPU-TEEs 
execute AI applications and the full-featured AI runtime inside the TEE, enlarging the trusted computing base 
(TCB). Type-II GPU-TEEs reduce the TCB by moving applications and complex runtimes outside the TEE and 
verifying execution via pre-recorded metadata. Existing lightweight Type-II designs target integrated GPUs, 
leaving cloud discrete GPUs (dGPUs) unprotected. We propose LGT4CG, a lightweight GPU-TEE for dGPUs. It 
includes GPU Shield, which secures CPU–GPU interactions, and Task Monitor, which verifies execution against 
metadata. We implement a prototype on ARM with NVIDIA RTX 3090. The runtime TCB consists of only about 
6K lines of C code. On DNNs, LGT4CG adds an average overhead of 4.16% for inference and 0.34% for training. 
On LLMs, the average overhead is 7.10% for TTFT and 6.05% for output throughput.
1. Introduction
AI applications execute diverse AI models, including Deep Neu
ral Networks (DNNs) and Large Language Models (LLMs), and have 
become increasingly powerful and widely adopted across numerous 
domains, including content generation, healthcare, finance, and be
yond [1–3]. An increasing number of organizations and individual 
developers choose to deploy their AI applications in the cloud to reduce 
management overhead and improve service quality. However, public 
cloud environments introduce a growing attack surface. In particular, 
security vulnerabilities in critical components such as hypervisors and 
GPU drivers have been repeatedly disclosed in recent years [4–6]. 
Under such threat models, adversaries with elevated privileges, includ
ing malicious cloud operators or co-resident tenants, may attempt to 
compromise AI applications by extracting sensitive data or tampering 
with GPU computations.
To run AI applications in untrusted clouds, researchers have pro
posed Trusted Execution Environments (TEEs) to ensure confidential
ity and integrity against untrusted system software and privileged 
users [7–12]. Since modern AI applications are typically accelerated 
by GPUs, GPU-TEEs have been developed to secure these applica
tions [13–23]. GPU vendors, e.g., NVIDIA, also provide proprietary 
GPU-TEE solutions [24]. As illustrated in Fig.  1, existing GPU-TEEs

2. Background
2.1. Characterizing AI software and GPU execution
AI software has a hierarchical structure, from top to bottom: AI 
application, the AI runtime, and the GPU driver. AI runtime refers to the 
entire user-space software stack on which an AI application depends. 
This stack includes AI frameworks such as PyTorch and vLLM, AI
specific libraries such as cuDNN, and the vendor-provided GPU runtime
such as CUDA runtime that sits underneath them.
GPUs are often shared among multiple users and applications, with 
different processes running concurrently on the same device. The GPU 
runtime manages a separate GPU context for each application. When 
an AI application accesses the GPU, it creates a GPU context through 
the GPU driver. A GPU context represents the logical execution envi
ronment of an AI task on the GPU and is used to isolate execution 
state and address space across different tasks. Within these contexts, 
high-level abstractions in applications such as operators, tensors, and 
computation graphs are translated into low-level execution primitives.
We classify execution primitives into four categories: Malloc, Free,
Copy, and Launch. Malloc primitives allocate and map memory re
gions in GPU memory, while Free primitives release these regions.
Copy primitives transfer input data and model weights into the allo
cated memory regions. Launch primitives initiate kernel execution on 
the GPU.
Integrated GPUs (iGPUs) iGPUs share system RAM with the CPU. 
In iGPU systems, GPU contexts are maintained entirely by software, 
as the iGPU hardware does not inherently track or enforce context 
isolation. The GPU runtime can directly update the GPU-accessible 
register that specifies the GPU page table to switch context. For Malloc
and Free, the GPU runtime completes GPU memory mapping and 
unmapping by updating the GPU page table in RAM. Similarly, Copy
primitives directly copy data into the allocated memory region in RAM.
Launch initiates kernel execution by writing the kernel entry address, 
arguments, and execution configuration directly into the corresponding 
GPU registers.
Discrete GPUs (dGPUs) dGPUs commonly interact with the CPU 
through the PCIe interconnect. During system boot, the BIOS and OS 
kernel enumerate PCIe devices and perform a series of configuration 
steps, including device identification, resource allocation, and Base 
Address Register (BAR) initialization. The GPU exposes its MMIO inter
faces and portions of VRAM by mapping them into BARs. These BAR 
regions establish the foundation for subsequent CPU–GPU interactions.
Unlike iGPUs, dGPUs do not realize execution primitives directly 
through registers, nor is context management implemented purely in 
software. GPU execution primitives on dGPUs are realized through 
hardware commands. Taking NVIDIA GPUs as an example, there are 
two types of commands. The first type consists of kernel launch com
mands for Launch. The second type consists of DMA commands for
Copy, Malloc, and Free. Copy transfers model weights and input 
data, while Malloc and Free transfer the GPU page tables. Context 
management is cooperatively handled by software and hardware. The 
software runtime is responsible for setting up and registering GPU 
contexts, whereas the hardware enforces context isolation and assists 
in scheduling and execution.
To support context management under this command-based execu
tion model, dGPUs introduce GPU channels. A channel is the abstrac
tion that allows the runtime to enqueue commands to the GPU. Each 
channel is associated with a command buffer1 and a ring buffer2.1. 
The runtime writes commands sequentially into the command buffer. 
Then it records the location and size of each command group in the 
ring buffer. The runtime then notifies the GPU by writing to a doorbell 
register exposed within BAR regions. After receiving the signal, the GPU 
fetches and executes commands from the command buffer. In addition, 
each GPU context maintains a DMA buffer in host memory that carries 
data for Copy commands. The driver establishes GPU mappings for the 
above buffers so that the GPU can read or write them.
2.2. GPU-TEE
The increasing migration of AI applications to cloud GPUs has 
amplified the risk of security breaches. Both the complexity and po
tential bugs in software [26], as well as the threat posed by malicious 
insiders [27,28], compromise the confidentiality and integrity of AI 
applications.
Trusted Execution Environments (TEEs) can create an isolated ex
ecution environment that protects applications from untrusted system 
software. TEEs have been widely used to protect various types of ap
plications. To protect AI applications, GPU-TEEs have been introduced 
to safeguard the confidentiality and integrity of GPU-accelerated AI 
applications. Both academia [13–22] and industry [24] have shown 
considerable interest in developing GPU-TEE solutions. We survey ex
isting GPU-TEEs, summarized in Table  1, and categorize them into

GPU-TEE design (Type-II).
Type-I Solutions The monolithic design integrates the entire AI 
runtime within the TEE to protect the input, model weights, kernel 
binaries, and execution of AI applications. The confidentiality and 
integrity of inputs, models, binaries, and execution are ensured through 
the inherent security features of TEEs. However, this design results in 
a large runtime TCB, as it requires porting the entire AI runtime into 
the TEE.
In addition, attackers may exploit software vulnerabilities within 
the AI runtime to compromise the integrity of computations. More 
than one hundred CVEs (Table  2) have been reported in modern 
AI software, which is fully incorporated into the TCB by all Type-I 
solutions. By exploiting vulnerabilities in these complex and error
prone software components, attackers can execute arbitrary code or 
manipulate sensitive state within the TEE, resulting in severe security 
risks.2
Type-II Solutions Lightweight approaches aim to address the is
sue of an excessively large TCB in Type-I designs by excluding AI 
applications and the bulky AI runtime from the TCB. To achieve this, 
Type-II GPU-TEEs adopt a two-stage execution model to protect the 
GPU computation of AI applications. Specifically, GPU computation 
is divided into a preparing stage and a running stage. The preparing 
stage is executed on a trusted platform to generate execution metadata, 
such as the computation graph and kernel binaries, for the target AI 
application. During the running stage, this metadata is leveraged to 
protect the online GPU computation of AI applications running on 
untrusted platforms. Different solutions utilize execution metadata in 
different ways. Some solutions [19,20] compare the metadata profiled 
offline with metadata generated online to verify the correctness of 
online GPU computation. Other solutions [21,22] adopt replay-based

Type-II GPU-TEE targeting dGPUs. Existing solutions such as Strong

mobile or edge platforms, which are not directly applicable to discrete 
GPUs widely deployed in cloud environments.
3. Overview
3.1. Goals
The primary goal of LGT4CG is to provide a lightweight and prac
tical GPU-TEE system capable of securely running GPU-accelerated AI 
applications on untrusted cloud dGPUs. The details are as follows:
•G1: Security: LGT4CG aims to provide strong security guar
antees for GPU execution, including isolating protected AI ap
plications from untrusted software and preventing unauthorized 
interference during execution. Simultaneously, LGT4CG ensures 
the correctness of AI application results.
•G2: Lightweightness: AI applications and the bulky AI runtime 
are excluded from the TCB. Only a minimal set of security-critical 
components is trusted, thereby significantly reducing the attack 
surface.
•G3: Compatibility: LGT4CG is implemented entirely using se
curity primitives available in existing hardware, requiring no 
custom hardware extensions. It can be integrated with existing 
AI software stacks with minimal modifications.
3.2. Threat model and assumptions
We assume a trusted preparing stage environment, e.g., a per
sonal laptop or workstation. All operations in the preparing stage 
are considered secure. Similar to prior Type-II GPU-TEE designs, we 
assume that users possess an identical GPU to that used in the online 
environment, enabling them to extract execution metadata offline. In 
contrast, we assume an untrusted running stage environment. Both co
located applications and system software in the running stage may be 
compromised by privileged attackers or controlled by malicious system 
administrators. These attackers may attempt to extract sensitive data 
from AI applications or tamper with computation results. We assume 
the local availability of suitable profiling and tracing tools to obtain

service (DoS) attacks, side-channel attacks, and physical attacks are also 
outside the scope of LGT4CG, which is consistent with prior GPU-TEE 
solutions.
3.3. LGT4CG architecture
As shown in Fig.  2, LGT4CG reuses the AI runtime in an untrusted 
environment and ensures safe GPU execution through two lightweight 
software components: GPU Shield and Task Monitor. They operate 
at a higher privilege level than the system software and intercept 
security-critical operations issued by both the GPU runtime and the 
GPU driver. GPU Shield establishes a secure and controlled context 
for transmitting GPU hardware commands, thereby guaranteeing the 
security of CPU–GPU interactions. Task Monitor is responsible for 
verifying the correctness of execution primitives, ensuring that online 
GPU execution conforms to the execution metadata captured during 
the preparing stage. With this architecture, LGT4CG leverages existing 
hardware isolation primitives to provide strong security guarantees 
(G1). Furthermore, LGT4CG excludes AI applications, the AI runtime, 
and other system software from the TCB, thereby achieving the design 
goal of lightweightness (G2). Finally, LGT4CG collaborates with the un
trusted AI runtime to verify critical operations rather than performing 
intrusive modifications, thereby ensuring compatibility (G3).
3.4. Workflow
The usage workflow of LGT4CG is illustrated in Fig.  3 and proceeds 
as follows.  
First, the user remotely attests the online LGT4CG GPU
TEE to establish trust. Upon successful attestation, the user sends a 
request for secure AI application execution to LGT4CG via a secure 
channel. The pre-generated execution metadata and cryptographic keys 
for protecting sensitive data are also transmitted at this stage.  Next, 
the user starts the AI application under the protection of LGT4CG. 
GPU Shield protects the context initialization for the AI application.

The encrypted data are then uploaded to the AI application. For each 
data transfer to the GPU, the untrusted AI runtime invokes LGT4CG 
through dedicated submission interfaces. Task Monitor decrypts and 
verifies the ciphertext inside the TEE, and then transfers the plaintext 
to the GPU.  The untrusted AI runtime generates execution metadata, 
such as a computation graph, and attempts to execute it on the GPU. 
Task Monitor verifies that the metadata generated conforms to the one 
prepared. If verification succeeds, the execution metadata is considered 
valid and secure and is submitted to the GPU for execution. Upon 
completion, the computation results are returned to the TEE, encrypted, 
and subsequently delivered to the GPU user.  Finally, the user sends 
a termination signal to the GPU-TEE over a secure channel. Once the 
AI application terminates, all sensitive data are securely erased and the 
GPU-TEE environment is safely destroyed.
4. Design
GPU Shield is responsible for constructing exclusive and trusted con
texts for AI applications. All interactions with the GPU are monitored 
by GPU Shield, preventing attackers from creating hidden contexts to 
bypass the protection. AI applications translate their workloads into 
hardware commands and submit them through the trusted context. 
Although attackers may attempt to submit malicious commands by 
compromising any part of the AI runtime, Task Monitor inspects all 
submissions and blocks the execution of malicious commands.
4.1. GPU shield
LGT4CG collaborates with the GPU driver to provide trusted con
texts for AI applications. To ensure context security, we implement a 
state machine within LGT4CG to track the execution status of each 
context: wait, running, and done. The contexts of an AI application pass 
through these three states sequentially and are never reused. LGT4CG 
protects context state transitions by intercepting and validating the 
critical context operations issued by the GPU driver. Any operation that

Context Creation When LGT4CG receives a new request for secure 
AI application execution, it creates a context status instance in the wait
state to track the AI application that is ready to run. This instance 
records not only basic identifiers, such as the associated context ID 
and the request source, but also the execution metadata required for 
subsequent secure execution. After the user launches the AI application, 
the untrusted GPU driver initiates the creation of context, channel, and 
buffer objects. These objects constitute the interface between the AI 
application and the GPU and can therefore serve as potential attack 
surfaces. LGT4CG uses hardware access control mechanisms to prevent 
the untrusted driver from directly creating these resources. Instead, the 
driver must redirect all creation requests to the TEE for validation. 
This ensures that even highly privileged attackers cannot bypass the 
protective mechanisms of LGT4CG to create malicious contexts.
For context creation, LGT4CG ensures that each created context 
has a one-to-one binding with a context status instance. Additionally, 
the address space configuration of the context must comply with the 
security policies of LGT4CG, which enforce isolation between different 
contexts. LGT4CG also ensures that the address spaces of the associated 
channel buffers are properly isolated to prevent malicious injection. 
After LGT4CG verifies all context-related resources, the corresponding 
context status instance transitions from the wait state to the running
state.
This transition indicates that the AI application is now ready to 
execute in a controlled and trusted environment. At this point, LGT4CG 
permits the AI application to submit commands to the context to per
form GPU computations. LGT4CG does not allow any reconfiguration 
or destruction of resources associated with a running context prior to 
receiving the termination signal from the GPU user.
Context Destruction Upon receiving a termination signal, LGT4CG 
sets the corresponding context status instance to the done state and 
disables further command submission on that context. After the AI 
application is terminated, the GPU driver issues a request to destroy 
the associated context. LGT4CG verifies the identity of the context and 
ensures that it is consistent with the record maintained in the context 
status instance before approving the destruction.
The destruction of a context triggers the cleanup of all related 
resources, including its channels and buffers. The bindings of these 
resources must remain consistent with the recorded metadata. Context 
destruction also involves sanitizing all GPU memory used under that 
context, which may include model weights, intermediate results, and 
input data. Since a context may be associated with multiple channels, 
buffers, and memory regions, LGT4CG not only enforces isolation be
tween different contexts but also ensures the completeness of resource 
cleanup. This prevents data leakage and eliminates residual data that 
could otherwise be exploited by an attacker.
4.2. Task monitor
When a context is in the running state, the AI application translates 
the AI model into execution primitives. These primitives are delivered 
to the GPU via the trusted context. Task Monitor is responsible for 
verifying these primitives. The primitive-level verification ensures the 
security and correctness of GPU execution.
For Malloc, LGT4CG enforces the following constraints. First, the 
physical memory spaces of different GPU contexts must be isolated. 
Second, no physical memory region can be mapped to multiple virtual 
address spaces within a single context. Third, mappings of sensitive 
data must reside in GPU VRAM rather than host RAM.
For Free, LGT4CG enforces a temporal constraint: the deallocation 
of GPU memory is disallowed until a task termination signal is re
ceived. This design is motivated by the observation that AI applications 
typically employ memory pooling strategies, in which GPU memory 
is allocated incrementally on demand, reused throughout the task 
lifetime, and reclaimed only upon task termination.
For Copy, LGT4CG provides end-to-end protection for sensitive data 
using cryptographic mechanisms. As for kernel binaries, only integrity 
protection is enforced, as kernel binaries are typically open-source and 
do not require confidentiality. As for input data and model weights, the 
owner encrypts them before submission. The corresponding keys are 
delivered to the TEE in advance through the secure channel established 
by remote attestation described in Section 3.4. At runtime, decryption 
and integrity verification are performed exclusively within LGT4CG. 
After successful decryption and verification, LGT4CG transfers the data 
to the GPU.
In addition, LGT4CG verifies the destination address and transfer 
size of the corresponding DMA command, thereby preventing mali
cious overwriting of model weights or input data used in subsequent 
computations. Once all checks pass, LGT4CG submits the DMA com
mand. Upon completion of the DMA operation, LGT4CG clears all 
CPU-side plaintext data and revokes access to the associated push
buffers and DMA buffers, ensuring that plaintext data do not persist 
in CPU-accessible memory.
For Launch, LGT4CG validates each launch command against cap
tured execution metadata. First, it verifies that the kernel entry address 
specified in the command points to an authorized kernel binary. Kernel 
binaries are transferred to the GPU via DMA. During this transfer, 
LGT4CG intercepts the DMA, computes a hash over the kernel binary 
to derive a unique identifier, and verifies its integrity. Within a GPU 
context, each kernel binary is typically transferred only once. For 
subsequent launch commands, LGT4CG only needs to verify the kernel 
entry address.
Beyond the kernel entry point and kernel binary, LGT4CG also 
verifies the kernel parameters carried in launch commands. For pointer
type parameters, LGT4CG verifies their input–output relationships. We 
use GPT-based AI applications as a representative example, illustrated 
in Fig.  4. The figure depicts a fragment of the computation graph for 
the Attention layer, together with the details of the first kernel. The 
first kernel is the RMS Normalization kernel ( ). It takes three pointer 
parameters, output, input, and weight, and three value parameters, ep
silon, num_tokens, and hidden_size. The output pointer designates the 
output buffer of the kernel. LGT4CG checks that this output pointer 
does not overwrite any buffer still required by later kernels. The input
pointer refers to the hidden state produced by a prior kernel. LGT4CG 
verifies that this pointer matches the output pointer of that prior kernel. 
The weight pointer refers to a slice of the model weights. LGT4CG 
verifies that this pointer is consistent with the buffer uploaded by Copy. 
For value-type parameters and configuration values, LGT4CG directly 
compares them against the expected values recorded in the execution 
metadata. Only exact matches are permitted for execution.
The subsequent kernels are validated under the same rule. By apply
ing this verification iteratively, each kernel consumes its parameters 
in an authorized manner, and the data flow across kernels remains 
consistent with the intended execution metadata. As a result, both the 
local correctness of individual kernels and the global correctness of 
inter-kernel dependencies are enforced.
5. Implementation

is equipped with a 64-core ARMv8 CPU running at 1.8 to 2.2 GHz. 
For GPU selection, we use NVIDIA products due to their widespread 
deployment in cloud services and strong software ecosystem support. 
Specifically, we use an NVIDIA RTX 3090 GPU (Ampere architecture), 
which features 82 computation units and is equipped with 24 GB 
VRAM. Our prototype runs Linux kernel v6.6.22, NVIDIA GPU kernel 
driver v515.76, CUDA v12.6, PyTorch v2.9.1, and vllm v0.6.0. The TCB 
of LGT4CG consists of approximately about 6K LoC. In addition, we

5.1. GPU shield
LGT4CG protects three classes of buffers used by the GPU: the per
GPU GSP message buffer, the per-channel command and ring buffers, 
and the per-context DMA buffer. These buffers reside in different 
physical locations and have different access patterns, but they share a 
common protection scheme: the TEE uses Stage-2 page tables to restrict 
CPU access, the IOMMU to restrict device DMA, and the TZASC to 
protect the page tables themselves. Stage-2 page tables are configured 
during trusted boot with identity mapping. IOMMU page tables are 
created on demand by the untrusted driver and verified by LGT4CG 
before activation. Subsequent modifications are validated by the TEE 
before being applied. Each buffer applies a tailored variant of this 
scheme, as detailed below.
GSP Communication Protection NVIDIA GPUs based on the Am
pere architecture and later integrate a dedicated GSP (GPU System 
Processor) chip on the GPU, which offloads part of the driver man
agement logic to the GPU side, including context management. The 
untrusted GPU driver transfers the GSP firmware to the GPU during 
context initialization. LGT4CG intercepts this process and redirects 
it into the TEE. During loading, LGT4CG checks the supplied image 
against the pre-recorded integrity hash. Only matching firmware is 
loaded. After loading, LGT4CG blocks any modification or overwrite 
on the firmware region during execution.
The GPU driver and the GSP interact through a producer–consumer 
model via a message buffer that resides in BAR2. The driver writes 
messages to the buffer and then triggers the GSP doorbell register, 
which resides in BAR0, to notify the GSP. We modify the driver to redi
rect GSP doorbell accesses to the TEE. Within the TEE, we inspect the 
GSP messages and record the results. During inspection, the associated 
memory regions are configured as read-only to prevent tampering, and 
the access permissions are restored once the inspection completes.
Channel Buffer Protection The NVIDIA GPU driver allocates a 
contiguous channel buffer of 16 MB, divided into chunks of 1 MB 
each, where each chunk contains a command buffer and a ring buffer 
(GPFIFO). The channel buffer also resides in the BAR region. LGT4CG 
inspects the GPU page tables of this buffer and constructs an equivalent 
mapping within the TEE that mirrors the GPU address translation. 
When a command is submitted, the TEE locates the corresponding 
ring buffer using the associated context ID and channel ID, resolves 
the command buffer through the mirror page table mappings, and 
verifies the command against the prepared execution metadata. The 
same read-only window scheme used for GSP messages applies during 
this verification. Once verification completes, the access permissions 
are restored.
DMA Buffer Protection The NVIDIA GPU driver selects a region 
in host RAM as the DMA buffer during execution. LGT4CG uses the 
DMA buffer to complete transfers of encrypted sensitive data. Before 
initiating a DMA, LGT4CG updates the page tables so that the DMA 
buffer is reachable only by the target dGPU and remains inaccessible 
to untrusted software and other peripherals. Once the DMA completes, 
LGT4CG clears the buffer contents and restores the default permissions.
Sensitive Data Upload LGT4CG exposes two interfaces to the AI 
runtime to upload the encrypted sensitive data. REGISTER copies the 
ciphertext of the request into the TEE. Upon REGISTER, the TEE de
crypts the ciphertext and verifies its integrity. SUBMIT asks the TEE 
to deliver specified requests. Upon SUBMIT, the TEE assembles the 
requested plaintexts in the protected DMA buffer, generates the cor
responding DMA command, and submits the command to transfer 
the data. The two interfaces above can conveniently replace existing 
unprotected data transfer interfaces. For synchronous decryption, both 
execute together after the prior GPU computation completes. For asyn
chronous decryption, REGISTER can complete when loading or upon 
receiving ciphertext data, while SUBMIT still executes after the prior 
GPU computation completes.
Context Creation The GSP uses an RPC API for creating (ALLOC_
OBJECT) objects. The API parameters specify the object type and 
its configuration. For example, the type identifier for a context is
CONTEXT_SHARE_A, while that for a channel is CHANNEL_GPFIFO_A.
For context creation RPCs, the driver transmits a UUID that uniquely 
identifies the GPU to the GSP. The GSP then returns a UUID repre
senting the newly created context, and the TEE records this identifier 
for subsequent verification. For AI applications executing on the same 
physical GPU, the GPU UUID remains identical, whereas the context 
UUIDs3 differ across applications. For channel creation RPCs, the driver 
transmits the GPFIFO pointer and the context UUID of the specific 
channel. The GSP then returns a UUID representing the newly cre
ated channel, and the TEE records the association among the context, 
channel, and GPFIFO.
Context Destruction The GSP uses an RPC API for releasing (FREE) 
objects. The FREE API requires only the ID of the object to be released. 
LGT4CG verifies that the ID matches the context being terminated, 
ensuring the correctness of the released resources. We observe that 
once the driver receives a context termination request, it first performs 
scrubbing over all memory regions associated with the context. After 
this sanitization, the driver issues the GSP RPC to free the context 
and its channels. To prevent potential data leakage, LGT4CG enforces 
protection over the scrubbing process. A context cleanup is consid
ered secure only if all associated memory regions have been properly 
scrubbed. If the driver attempts to free the context before scrubbing is 
complete, LGT4CG treats this as a security violation and reports it to 
the GPU user.
PCIe Configuration Access Control After PCIe enumeration com
pletes, information about resource allocation and BAR mappings is 
stored in the PCIe configuration space. Although secure boot protects 
these configurations during the boot process, improper modifications at 
runtime can undermine GPU isolation. For example, an attacker could 
maliciously modify BAR registers to remap MMIO regions to unau
thorized physical addresses. To address this threat, LGT4CG applies a 
blocking policy that prevents untrusted CPU software and untrusted 
devices from arbitrarily modifying the PCIe configuration space. Specif
ically, all write accesses to the PCIe configuration space are denied 
after secure boot, ensuring that the established GPU resource mappings 
cannot be tampered with at runtime.

5.2. Task monitor
We redirect the transmission of hardware commands to the TEE by 
directly modifying the source code of the kernel driver and applying

construction. When verifying hardware commands, we apply the same 
memory protection scheme used for GSP message protection to ensure 
the integrity of the verification process.
6. Security analysis
Due to limitations of CPU firmware, certain components of our 
design require support from the platform vendor. We believe that 
this requirement is practical and can be readily addressed by GPU or 
platform vendors. In particular, we expect LGT4CG to run at EL3, which 
provides the highest privilege level on ARM platforms and is well suited 
for enforcing strong isolation and secure mediation of GPU resources. 
LGT4CG itself is fully trusted and assumed to be free of bugs. We next 
summarize representative attacks that an adversary may mount from 
outside the TEE.
Attackers may attempt to load a malicious GSP firmware image 
to undermine the protection of LGT4CG. The firmware integrity ver
ification inside LGT4CG checks the firmware before GPU execution 
(Section 5). A tampered image fails this verification.
Attackers may attempt to access GPU VRAM directly from the CPU 
through PCIe BAR windows, or tamper with BAR registers in the PCIe 
configuration space to enlarge the exposed range. LGT4CG runs at 
EL3 and exclusively manages the Stage-2 translation, keeping BAR
exposed physical ranges unmapped in the normal world. It also blocks 
all writes to the PCIe configuration space after secure boot, preventing 
runtime BAR remapping. Both mechanisms together defend against this 
BAR-based attack.
Attackers may attempt to redirect the GPU mappings of sensitive 
data to host RAM, where the CPU can bypass the protection of LGT4CG 
and read the data directly. LGT4CG verifies every GPU page table and 
rejects RAM mappings for sensitive data.
Attackers may attempt to forge malicious commands to transfer sen
sitive data from the GPU to host RAM. To mitigate this threat, LGT4CG 
ensures that computation results are encrypted upon completion, and 
prevents any GPU memory region from being returned to or directly 
accessed from host memory.
Attackers may control other devices to launch DMA attacks.
LGT4CG addresses this threat by inspecting the IO page tables during 
GPU context creation to ensure that no address space is shared between 
the GPU and other devices. All modifications to the IO page tables must 
be verified by LGT4CG to prevent unauthorized devices from accessing 
GPU memory during computation.
Attackers may attempt to launch cross-application attacks by ex
ploiting the context of other applications to compromise protected AI 
applications. To mitigate this risk, LGT4CG enforces strict context iso
lation between different AI applications. Specifically, LGT4CG records 
the associations among contexts, channels, and pushbuffers, ensuring 
that every hardware command is validated against the correspond
ing execution metadata of its owning context. In addition, LGT4CG 
configures separate GPU page tables for different contexts, preventing 
memory mappings from being shared across applications and thereby 
eliminating cross-context memory interference.
Attackers may attempt to modify kernel binaries to subvert AI 
applications. LGT4CG detects such modifications through kernel binary 
verification. LGT4CG tracks the lifecycle of each kernel and monitors 
DMA commands, ensuring that verified kernels cannot be maliciously 
altered.

7.1. Experimental setup
We selected eleven AI models to evaluate the end-to-end perfor
mance overhead introduced by LGT4CG. The DNN models include 
ResNet [33] (resnet152 [34]), DenseNet [35] (densenet201 [36]), 
VGG [37] (vgg19 [38]), ViT [39] (vit_l_32 [40]), EffNet [41] (ef

the ImageNet dataset. The LLM models include Llama [45] (Llama

2.7B [50]), Qwen [51] (Qwen2.5-1.5B [52]), and GPT [53] (GPT

7.4. Graph mode support
Modern GPUs support graph execution mode, which means launch
ing a group of kernels within a single graph-execute command, elimi
nating the kernel launch overhead. On the first graph execution, the AI 
runtime dispatches kernel launch commands one by one. These com
mands are stored contiguously in a dedicated VRAM region. LGT4CG 
verifies each command as described in Section 4.2 and protects this 
VRAM region during the graph execution. On subsequent graph execu
tions, the AI runtime sends a single graph-execute command pointing 
to the start of the captured graph. LGT4CG verifies this command and 
confirms that it targets the captured graph. This scheme further reduces 
the verification overhead.
We use the LLM inference as a representative case and rerun the 
same experiments under graph mode. Fig.  12 shows that graph mode 
runs faster and incurs smaller overhead.
8. Discussion
8.1. Design selection
As discussed above, Type-II GPU-TEEs can be implemented in 
two different ways: verification-based and replay-based approaches. 
LGT4CG adopts a verification-based approach rather than a replay
based one. This is because, for dGPUs, the replayer inside the TEE 
must perform context management and resource allocation indepen
dently, which significantly increases implementation complexity. In 
contrast, verification focuses only on the critical operations performed 
9

Journal of Systems Architecture 178 (2026) 103886 
by the driver and runtime. By validating key context management 
operations together with GPU execution primitives, LGT4CG main
tains a lightweight design while preserving compatibility with existing 
software stacks.

Moreover, verification does not sacrifice performance. Modern GPU 
vendors, such as NVIDIA and AMD, provide support for graph-based 
execution models (e.g., CUDA Graphs and HIP Graphs). Graph-based 
execution models are in essence analogous to replay-based execution. 
Verification can be extended to cover the graph to be executed, thereby 
achieving performance comparable to that of replay-based approaches.
8.2. Execution metadata collection
Execution metadata, such as kernel parameters and binaries, plays 
a crucial role in verifying the behavior of online AI applications. 
However, the focus of this work is not on the collection mechanisms or 
the detailed structure of execution metadata itself. Consequently, we 
assume that such metadata is available through standard profiling or 
runtime instrumentation mechanisms provided by modern GPU tools 
(e.g., CUDA or HIP). We treat it as an auxiliary input to the GPU-TEE 
system.
8.3. Offline input processing
We assume that the AI application does not require additional 
CPU-side logic to preprocess input data at runtime, such as image 
transformations or tokenization. This assumption is reasonable because 
such preprocessing can be performed offline prior to execution, or 
alternatively offloaded to the GPU, where it can be executed more 
efficiently. Including input transformation logic within the TEE would 
not only unnecessarily enlarge the TCB, but also introduce significant 
engineering complexity.
8.4. Dynamism support
AI applications inherently exhibit dynamism during execution. We 
identify two sources of dynamism with different causes. Input-shape 
dynamism arises from shape dependence: different input shapes lead 
the runtime to select different kernel implementations for the best 
performance. Control-flow dynamism arises from content dependence: 
different content in the inputs or intermediate results steers the AI 
application into different branches.
Input-shape dynamism. For best performance, the AI runtime per
forms kernel selection per launch based on the input shape. For example, 
the runtime picks different GEMM tile sizes or Winograd variants for 
different convolution shapes. This selection depends only on the shape 
and is independent of tensor contents.
LGT4CG therefore verifies each launch against pre-recorded meta
data indexed by input shape. In the offline preparing stage, LGT4CG 
enumerates the admissible input shapes and records the details of each 
kernel. At runtime, LGT4CG looks up the kernel set for the reported 
shape and admits a launch only when every parameter matches.
For AI applications using DNNs, the model fixes every dimension 
except batch size. DNN training typically uses a single batch size 
throughout, so LGT4CG records one metadata variant. DNN inference, 
in contrast, may serve requests with different batch sizes, so LGT4CG 
records one metadata variant per admissible batch size.
For AI applications using LLMs, we focus on the inference scenario. 
LLM inference involves more dimensions, such as batch size, prompt 
length, and the number of generated tokens. These dimensions pro
duce an explosion of combinations that is impractical to enumerate 
offline. However, we observe that only two factors actually drive 
kernel selection: batch size and the number of tokens in the batch 
(total_tokens), and their influence varies across the forward pass. 
Attention and sampling kernels depend on batch size, while the other 
kernels depend only on total_tokens. LGT4CG therefore collects

metadata by emulating different factors for each part of the forward 
pass.
Control-flow dynamism. The current prototype does not support 
applications with dynamic control flow such as early exits or data
dependent branches. We acknowledge this as a limitation of the present 
design, and we leave support for future work.
9. Related work
9.1. GPU-TEE
There is growing interest in GPU-TEEs from both academia and 
industry [13–22,24,56,57]. We categorize existing solutions into two 
types. Type-I GPU-TEE solutions [13–18] are hindered by a large TCB. 
Type-II GPU-TEE solutions [19–22] leverage pre-captured execution 
metadata to protect AI applications. However, there is currently no 
reference implementation of a lightweight Type-II GPU-TEE target
ing dGPUs. This paper presents LGT4CG, a lightweight and practical 
GPU-TEE system for dGPUs.
9.2. GPU attacks
GPU attacks can be categorized into hardware-based physical at
tacks and software-based attacks. Researchers have proposed various 
memory encryption schemes to defend against physical attacks [58
63]. Software attacks against GPUs exploit buffer overflows to execute 
malicious functions [64–66]. LGT4CG is orthogonal to the above at
tacks and solutions, and can be combined with existing approaches to 
defend against a broader range of potential attacks.
9.3. AI attacks
In recent years, the security of deep learning has attracted con
siderable attention from researchers and practitioners. Prior work has 
explored potential attacks against AI tasks as well as corresponding 
defense techniques [67–69]. To prevent adversaries from extracting 
private information, AI computations often incorporate add-on com
ponents such as anomaly detection modules and adversarial training 
networks. LGT4CG protects the integrity of every stage of AI execution, 
and the above defenses against AI attacks remain fully compatible with 
our approach.
10. Conclusion
This paper analyzes existing GPU-TEE systems and categorizes them 
into Type-I and Type-II designs. Type-I GPU-TEEs incur a large TCB 
that encompasses the entire AI runtime. Type-II systems execute AI 
applications in two phases, removing the AI runtime from the runtime 
TCB. The first phase generates execution metadata, and the second 
phase verifies GPU execution against the metadata generated offline. 
However, existing Type-II GPU-TEE implementations have primarily 
targeted mobile or edge platforms with integrated GPUs (iGPUs). Dis
crete GPUs (dGPUs), which dominate cloud computing environments, 
have received little attention in the design of lightweight GPU-TEEs. To 
bridge this gap, we design LGT4CG, a lightweight GPU-TEE specifically 
tailored for dGPUs. We implement a prototype of LGT4CG on an ARM
based platform with an NVIDIA RTX 3090 GPU, and the runtime TCB 
of LGT4CG is limited to approximately about 6K lines of C code. We 
evaluate the performance of LGT4CG using widely adopted AI models. 
The results show an average overhead of 4.16% for DNN inference and 
0.34% for DNN training. On LLMs, the average overhead is 7.10% for 
TTFT and 6.05% for output throughput.
